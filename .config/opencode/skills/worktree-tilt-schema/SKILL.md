# Worktree Tilt Schema Regeneration

Temporarily modify a Tiltfile in a git worktree so the NestJS service can boot locally and regenerate `schema.gql`.

## Problem

Git worktrees live under `.claude/worktrees/` (or similar), so the Tiltfile's relative `../` upstream includes resolve to the wrong directory. Upstream Tiltfiles also form circular chains that re-include the current repo's Tiltfile, causing `local_resource named "X" already exists` errors.

## When to use

- You changed `@Field` decorators (e.g. adding `{ nullable: true }`) and need to regenerate `schema.gql`.
- CI's smoke test (`test/smoke/schemaDiff.spec.ts`) fails because `schema.gql` is stale.
- You are in a git worktree and `tilt up` fails with resource-already-exists errors.

## Important rules

- **Never edit schema.gql directly.** It must be regenerated by booting the NestJS app.
- **Never commit the Tiltfile changes.** Revert them after schema generation.
- **Stop any existing Tilt session** from the main repo first (`tilt down` from the main repo), or run on a different port (`tilt up --port 10351`).

## Workflow

### Step 1: Detect the worktree and main repo

```bash
# Get worktree path and main repo path
WORKTREE=$(pwd)
MAIN_REPO=$(git worktree list | head -1 | awk '{print $1}')
DEV_DIR=$(dirname "$MAIN_REPO")
```

### Step 2: Modify the Tiltfile

Comment out all `mryum_v1.include_upstream(...)` lines. These pull in sibling repos that eventually circle back and re-declare the same resources.

Keep the local `include('./tiltfile.d/...')` lines intact, as those define the actual service resources (manage-api, manage-worker, etc.).

**Before:**
```python
mryum_v1.include_upstream("../menu-api/Tiltfile")
mryum_v1.include_upstream("../mr-yum/dev-env/Tiltfile")
# ... more upstream includes ...
include('./tiltfile.d/manage-api.star')
include('./tiltfile.d/manage-worker.star')
```

**After:**
```python
# TEMPORARY: upstream includes disabled for worktree schema generation
# mryum_v1.include_upstream("../menu-api/Tiltfile")
# mryum_v1.include_upstream("../mr-yum/dev-env/Tiltfile")
# ... more upstream includes commented out ...
include('./tiltfile.d/manage-api.star')
include('./tiltfile.d/manage-worker.star')
```

### Step 3: Ensure dependencies are running

The manage-api service depends on external services (CockroachDB, Redis, Redpanda, etc.) that are normally started by the upstream includes. These must already be running from the main repo's Tilt session or from docker-compose.

Tell the user: "Make sure your infrastructure services (DB, Redis, Redpanda) are already running from the main repo or docker-compose before running tilt up here."

### Step 4: Boot the service

Tell the user to run:

```bash
tilt up
```

If they get port conflicts with an existing Tilt session:

```bash
tilt up --port 10351
```

Wait for the manage-api service to be ready (check `http://127.0.0.1:5010` or the port configured in the service).

### Step 5: Verify schema.gql was regenerated

```bash
git diff -- schema.gql
```

The diff should show the expected changes (e.g. `Boolean!` becoming `Boolean` for newly nullable fields, or `String!` becoming `[String!]!` for array fields).

### Step 6: Commit schema.gql only

```bash
git add schema.gql
git commit -m "chore: regenerate schema.gql for nullable field changes"
```

### Step 7: Revert the Tiltfile

```bash
git checkout -- Tiltfile
```

### Step 8: Stop the worktree Tilt session

Tell the user to press Ctrl+C in the Tilt terminal, or run:

```bash
tilt down
```

## Troubleshooting

### `local_resource named "X" already exists`

The upstream includes are circling back into the current repo. Make sure ALL `include_upstream` lines are commented out.

### Service fails to connect to DB/Redis/Redpanda

The infrastructure services aren't running. Start them from the main repo's Tilt session or docker-compose first.

### schema.gql not changing

The NestJS app may not have booted successfully. Check the Tilt UI or logs for errors. The schema is generated on startup by the `@nestjs/graphql` autoSchemaFile option.
